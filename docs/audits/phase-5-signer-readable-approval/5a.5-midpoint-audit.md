# Phase 5A.5 Midpoint Audit

## Scope

Reviewed Phase 5A Signer Payload and Local Approval Scaffold through the local signer boundary.

In scope:

- Generic signer adapter contract and claim-level vocabulary.
- Deterministic EIP-712 typed-data mapping from canonical `AgentIntent`.
- ClearIntent app/CLI approval preview field coverage.
- Deterministic fixture signature evidence compatible with core signature evidence.
- Wallet class, display status, blind/limited-display warning vocabulary.
- Conditional review prompt semantics for value threshold, high risk, new executor, degraded audit write, and policy change.
- Center CLI signer inspection route if implemented by Agent B.

Out of scope:

- Real MetaMask, WalletConnect, mobile, smart-account, or hardware wallet signing.
- Wallet-rendered typed-data preview claims.
- Secure-device display claims.
- Vendor-approved Clear Signing claims.
- Phase 2B live 0G, Phase 3B live ENS, Phase 4B live KeeperHub/onchain execution, Guardian Agent, dashboard, deploy, or hosted app work.

## Evidence

- `packages/signer-adapter/src/types.ts` defines local signer claim levels, wallet classes, capability levels, display status, display warnings, signer issue codes, approval field map, fixture signature evidence, and review prompt types.
- `packages/signer-adapter/src/eip712.ts` builds deterministic ClearIntent EIP-712 typed data from canonical `AgentIntent` with explicit `chainId` and `verifyingContract`.
- `packages/signer-adapter/src/preview.ts` renders the ClearIntent approval preview from the same authority-critical field map.
- `packages/signer-adapter/src/fixture-signer.ts` creates deterministic local fixture signature evidence and converts it to core-compatible signature evidence.
- `packages/signer-adapter/src/review-prompts.ts` implements conditional review prompt semantics.
- `packages/signer-adapter/src/injected-wallet.ts` prepares only `eth_signTypedData_v4` request shape and typed provider issue-code mapping.
- `tests/signer-adapter/signer-adapter.test.ts` covers typed data, preview, fixture signature evidence, conditional review prompts, metadata, and injected-wallet request shape/status.
- `packages/center-cli/src/signer-status.ts` exposes signer status/preview/typed-data/metadata routes while normalizing all claims to local-only inspection.

## Boundary Review

- Claim levels must be limited to `signer-local-fixture` and `eip712-local-fixture`.
- A ClearIntent approval preview is app/CLI-rendered evidence only.
- A fixture signature is deterministic local evidence only.
- EIP-712 typed data is signing payload evidence, not proof that a wallet displayed or signed it.
- Signing must not replace `HumanReviewCheckpoint`; the explicit human intervention gate remains contract truth.
- Missing wallet display evidence must surface as `unknown`, `app_preview_only`, or `blind_or_limited_display`, not as success.

## Required Field Coverage

The local preview and typed payload must expose or derive these authority-critical fields when available:

- intent id/hash
- policy hash
- action hash
- signer
- executor
- nonce
- deadline
- chain id
- verifying contract
- value bounds
- identity references
- audit references

## Findings

- Critical: none in Agent C docs scope.
- High: none.
- Medium: none.
- Low: local fixture signatures can be mistaken for wallet support unless audits and docs keep the claim level explicit.
