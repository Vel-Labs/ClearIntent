# Closeout Audit: Phase 1.5 Center CLI Skeleton

## Scope

Phase 1.5 builds the local Center CLI skeleton over `packages/core` before provider adapters or demo integration begin.

Changed in scope:

- added `packages/center-cli/` as the local Center CLI package
- added repo-local npm scripts for CLI usage
- added command families:
  - `clearintent center status`
  - `clearintent center inspect`
  - `clearintent intent validate`
  - `clearintent intent state`
  - `clearintent authority evaluate`
  - `clearintent module list`
  - `clearintent module doctor`
- added fixture-backed command behavior from `contracts/examples/`
- added human-readable terminal output as the default
- added explicit `--json` output for deterministic agent/automation parsing
- added local module registry and doctor skeleton with future adapters marked `deferred`
- added a bare `npm run clearintent` human landing path and interactive TTY menu wrapper
- added explicit fixture metadata to human and JSON output: `mode`, `fixtureSource`, and `liveProvider`
- added tests for command routing, output modes, blocked/missing evidence, authority issue codes, and module doctor behavior
- updated architecture, boundary, roadmap, current-state, decisions, changelog, repo profile, and file-tree routing

Stayed out of scope:

- ENS implementation
- 0G storage or compute integration
- KeeperHub execution
- Ledger, hardware signer, or wallet signer integration
- webhook delivery
- OS notification delivery
- browser UI or hosted service
- demo integration
- provider adapter directories
- `contracts/` changes

## Exact Results

- `npm test -- tests/center-cli`: passed. 9 tests passed in `tests/center-cli/center-cli.test.ts`.
- `npm run clearintent -- center status`: produced human-readable status showing blocked execution, lifecycle `human_approved`, next action `collect_signature`, and missing evidence `signature`.
- `npm run clearintent -- center inspect --json`: emitted command-level JSON with no CLI prose; when run through npm, use `npm run --silent` for parse-safe consumption without npm's banner.
- `npm run clearintent -- authority evaluate --fixture expired --json`: emitted command-level JSON with fail-closed issue codes `deadline_expired` and `deadline_before_created_at`.
- `npm run clearintent -- module doctor`: reported core `ready`; ENS, 0G, KeeperHub, signer, and notifications remain `deferred`.
- `npm run typecheck`: passed.
- `npm run validate:scaffold`: passed. Scaffold profile paths exist, commands resolve, forbidden early-scope directories are absent, `docs/FILE_TREE.md` is current, and template placeholders are registered.
- `npm run validate:contracts`: passed. Valid fixtures matched schemas, invalid fixtures were rejected or semantically flagged, and review/audit hashes bind correctly.
- `npm run test:contracts`: passed. 9 tests passed in `tests/contracts/validate-contracts.test.ts`.
- `npm test`: passed. 39 tests passed across 5 test files.
- `npm run check`: passed. It ran scaffold validation, contract validation, contract tests, all tests, and TypeScript checking successfully.
- `git diff --check`: passed with no whitespace errors.
- `git status --short`: showed only scoped Phase 1.5 code, tests, audit, and routing/doc changes.
- `npm run --silent clearintent -- center inspect --json`: emitted JSON as the first byte of stdout; command exited nonzero because the fixture is intentionally blocked on missing signature evidence.

## Findings

- Critical: none.
- High: none.
- Medium: none.
- Low: This phase proves local CLI skeleton behavior only. It does not prove live provider integration, live signer behavior, webhook/OS notification delivery, hosted UI, or demo execution.
- Low: `npm run` emits npm's wrapper banner unless `--silent` is used. Machine consumers should use `npm run --silent clearintent -- <command> --json` or call the CLI entrypoint in-process.
- Low: Blocked authority results are data, not command failures. Read-only inspection and evaluation commands now exit `0` when rendering succeeds and expose separate `commandOk` and `authorityOk` fields. `ok` remains a compatibility alias for `authorityOk` during the skeleton phase.

## Follow-Up

Provider adapter scaffolding may start next through the Center/module boundary. The recommended next workstream should be one bounded adapter package at a time, with no adapter redefining lifecycle, review, signer, executor, deadline, audit, or authority rules already owned by `contracts/` and `packages/core`.
