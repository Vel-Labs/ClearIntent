# Decisions: 2026-04-26

## Contract/core stability handoff gates downstream work

- Status: Accepted
- Context: ClearIntent can only parallelize provider adapters and demo implementation safely after canonical authority truth and executable enforcement are stable.
- Decision: Treat `contracts/` plus `packages/core/` as the final stability handoff before ENS, 0G, KeeperHub, signer, Ledger, or demo work proceeds.
- Consequences: Future scaffolds should include this pattern by default when correctness, replayability, or multi-agent implementation matter. Adapters and demos must consume the contract/core API rather than becoming alternate sources of truth.

## Root roadmap and decisions are indexes

- Status: Accepted
- Context: ClearIntent needs a quiet root directory without losing high-visibility surfaces that humans, judges, and agents expect.
- Decision: Keep root `ROADMAP.md` and `DECISIONS.md` as indexes and summaries. Put detailed roadmap scope under `docs/roadmaps/` and dated decision entries under `docs/decisions/`.
- Consequences: `REPO_PROFILE.json`, `README.md`, `AGENTS.md`, and `docs/agents/START_HERE.md` must route agents to the detailed docs. Do not duplicate full decision truth between root and dated files.

## Core API is callable before transport-specific APIs

- Status: Accepted
- Context: ClearIntent will need CLI, webhook, notification, and possibly HTTP or MCP surfaces, but authority behavior must not be defined by ambiguous transport verbs.
- Decision: Treat `packages/core/API.md` as the callable authority API for Phase 1C and require future transport layers to document how they wrap core functions.
- Consequences: CLI, webhook, OS notification, HTTP, MCP, and GUI layers must preserve core lifecycle and verification semantics. They may add transport contracts, but they must not redefine policy, review, signature, execution, or audit authority.
