# Decisions: 2026-04-27

## Wallet-neutral signer strategy

- Context: Ledger secure-screen Clear Signing for the demo path depends on provider approval that may not be available in time. ClearIntent still needs a strong signing story without becoming a single-wallet product.
- Decision: Treat signer support as wallet-neutral. ClearIntent supports clear-signing-ready EIP-712 typed intents and ClearIntent-rendered human-readable approval metadata across injected wallets, WalletConnect wallets, smart accounts, and hardware-backed wallets. Wallet-rendered previews, secure-device display, and vendor-approved Clear Signing are reported separately by tested capability level.
- Consequences: Software wallets and WalletConnect become first-class integration targets. Hardware wallets remain important assurance layers, with Ledger official secure-screen Clear Signing treated as a provider approval and certification path rather than a baseline demo claim.

## Progressive wallet demo

- Context: The demo needs to communicate both accessibility and stronger signer-side assurance without overstating unapproved Clear Signing.
- Decision: The preferred demo shows the same ClearIntent intent across app preview, software wallet signing, WalletConnect where practical, and hardware wallet display where available. The audit bundle should prove each path maps back to the same intent hash, policy hash, review checkpoint, and signer result.
- Consequences: Demo scripts and submission checks must distinguish ClearIntent app preview, wallet-rendered preview, secure-device display, and vendor-approved Clear Signing.

## Center CLI product boundary

- Context: Phase 1 closed the contract/core authority kernel, but future adapters needed one operator-facing product center before ENS, 0G, KeeperHub, signer, notification, or demo work began.
- Decision: `packages/center-cli/` is the local Center CLI skeleton. It consumes `packages/core` public primitives, renders human-readable terminal summaries by default, and emits deterministic JSON only through explicit `--json` usage. When invoked through npm for machine parsing, use `npm run --silent clearintent -- <command> --json` so npm output does not precede the JSON payload.
- Consequences: Provider adapter work may start after Phase 1.5, but adapters must plug into this Center/core boundary and must not redefine lifecycle, review, signer, executor, deadline, audit, or authority semantics.
